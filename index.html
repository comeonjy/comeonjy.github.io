Protobuf编码格式

编码方式：HEAD1+MSG1+HEAD2+MSG2……
HEAD=(field_num << 3) | wire_type

field_number：字段序号，可以任意顺序使用但不能重复
wire_type：数据类型

类型编号 类型名 pb类型
0 Varint int32、int64、uint32、uint64sint32、sint64、bool、enum
1 64-bit fixed64、sfixed64、double
2 Length-delimited string、bytes、嵌入式类型、打包的重复字段

MySQL事务与并发控制

发表于 2021-03-17 分类于 MySQL Valine：
本文字数： 636 阅读时长 ≈ 1 分钟

事务

ACID

  • 原子性：一个事务必须被视为一个不可分割的最小工作单元，整个事务中的整个操作，要么全部成功提交，要么全部失败回滚

    实现原理：undo log（回滚日志）

    属于逻辑日志，记录事务SQL执行的相关信息，一旦发生回滚，根据InnoDB的undo log做与之相反的操作

  • 一致性：数据总是从一个一致性状态转换为另一个一致性状态

  • 隔离性：一个事务所做的修改在最终提交前，对其他事务是不可见的

    • 读未提交：一个事务可以读取到另一个事务未提交的修改（脏读）
    • 读已提交：一个事务只能读取到另一个事务已经提交的修改（不可重复读）
    • 可重复读：事务只能读取在它开始之前已经提交的修改，同一事务多次读取相同的数据返回结果一样（默认）
    • 串行化：事务串行化

  • 持久性：事务一旦提交，其所作的修改就会永久保存在数据库中

    实现原理：redo log（重做日志）

    物理格式的日志，记录的是物理数据页面的修改信息

索引概述

索引设计

  • 数据结构尽量简单，主要是数据之间的比较
  • 尽量不为空，索引不会包含有空值的列
  • 前缀索引，指定索引长度
  • 避免or条件，如果or条件前面部分使用索引后面没用，则会放弃所有索引
  • 在列上进行运算或以通配符开头的like不会使用索引

最左前缀原则

  • 查询必须从索引最左边开始，且不能跳过
  • 不能使用范围查询右边的列

  • 遍历灰色集合，将其引用的对象置灰，自身置黑
  • 重复上一步骤，直至无灰色对象

Git Flow 分支管理

  • master：主分支（用于版本发布，始终与线上一致）
  • dev：开发分支（用于开发，提测时，从dev检出release-1.0.0分支）
  • release: 预发布（用于测试，测试中有问题直接修改，测试完成后合并入master和dev）
  • feature-*：功能分支（用于功能开发，完成后合并到dev）
  • hotfix-*：修复bug（从master分出来，完成后合并到master和dev）
